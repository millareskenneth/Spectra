## Core Concepts — Understand These Before Writing Any Code

### 1. Traffic Capture (No Root)
- Android: `VpnService` creates a TUN interface. All device traffic routes through it.
  The app acts as a local MitM proxy.
- iOS: `NEAppProxyProvider` intercepts app-level traffic. Requires the
  `com.apple.developer.networking.networkextension` entitlement.
- Both platforms: User installs a custom CA certificate generated by the app.
  The proxy dynamically signs per-domain certs with this CA to decrypt HTTPS.

### 2. SSL/TLS Interception
- App generates a self-signed CA on first launch (stored securely in Keychain/Keystore)
- For each HTTPS connection: generate a leaf cert for that domain, sign with our CA
- Forward decrypted traffic to the real server using the real cert chain
- Apps with **certificate pinning** will reject connections — flag these as
  `PINNING_DETECTED` in the UI, do not try to bypass silently

### 3. Security Rule Engine
Every captured `RequestResponsePair` passes through the rule engine synchronously.
Rules are pure functions: `(req, res) => RuleMatch | null`
Rules must NEVER throw — wrap logic in try/catch and return null on error.
Sensitive values in `evidence` fields MUST be masked before storage or display.

### 4. Data Model
```typescript
interface CapturedRequest {
  id: string;
  timestamp: number;
  method: string;
  url: string;
  headers: Record<string, string>;
  body: string | null;
  packageName?: string;      // Android only
  bundleId?: string;         // iOS only
}

interface CapturedResponse {
  statusCode: number;
  headers: Record<string, string>;
  body: string | null;
  durationMs: number;
  bodySize: number;
}

interface SecurityFinding {
  ruleId: string;            // e.g. "REQ-001"
  severity: 'critical' | 'high' | 'medium' | 'low' | 'info';
  category: 'exposure' | 'transport' | 'headers' | 'data-leak' | 'auth' | 'jwt' | 'cookie';
  name: string;
  message: string;
  evidence: string;          // ALWAYS masked
  recommendation: string;
}
```

## Security Rule Catalogue

When writing or modifying rules, follow this numbering scheme:

| ID Range | Category |
|---|---|
| REQ-001 to REQ-099 | Request-side issues |
| RES-001 to RES-099 | Response-side issues |
| HDR-001 to HDR-099 | Header issues |
| JWT-001 to JWT-099 | JWT issues |
| COK-001 to COK-099 | Cookie issues |
| TLS-001 to TLS-099 | Transport/TLS issues |

### Currently Defined Rules

**REQ-001** — API key or secret in URL query parameter (critical)
**REQ-002** — Password in URL query parameter (critical)
**REQ-003** — AWS credential pattern detected (`AKIA...`) in request (critical)
**REQ-004** — Authorization token sent over plain HTTP (critical)
**REQ-005** — Private key pattern in request body (critical)

**RES-001** — Stack trace in response body (high)
**RES-002** — Internal server paths exposed in response (high)
**RES-003** — Auth endpoint returns sensitive fields (password, hash, salt, ssn) (high)
**RES-004** — Response contains excessive user PII beyond auth scope (high)
**RES-005** — DB internals leaked (raw SQL errors, ORM field names, sequence IDs) (medium)
**RES-006** — Server technology fingerprint in response headers (medium)
**RES-007** — Response returns full DB row on login/profile endpoints (high)

**HDR-001** — Missing Strict-Transport-Security on HTTPS response (medium)
**HDR-002** — Missing Content-Security-Policy (medium)
**HDR-003** — Missing X-Content-Type-Options (low)
**HDR-004** — Missing X-Frame-Options (low)
**HDR-005** — X-Powered-By or Server header reveals stack (info)

**JWT-001** — JWT with alg: none (critical)
**JWT-002** — JWT missing exp claim (high)
**JWT-003** — JWT contains sensitive PII in payload (high)
**JWT-004** — JWT sent over plain HTTP (critical)

**COK-001** — Session cookie missing Secure flag (high)
**COK-002** — Session cookie missing HttpOnly flag (high)
**COK-003** — Cookie missing SameSite attribute (medium)

**TLS-001** — HTTPS endpoint falls back to HTTP redirect (high)
**TLS-002** — Certificate pinning detected (info — not a vulnerability, just flagged)

## Coding Standards

### TypeScript
- Strict mode always on (`"strict": true` in tsconfig)
- No `any` types — use `unknown` and narrow explicitly
- All async functions must handle errors — no unhandled promise rejections
- Sensitive data (keys, tokens, passwords) must never be logged to console

### Native Code
- Kotlin: use coroutines for async VPN/proxy operations
- Swift: use async/await, never force-unwrap optionals
- Native modules must emit events to RN thread-safely

### Security Rule Functions
```typescript
// CORRECT — safe rule pattern
const myRule: SecurityRule = {
  id: 'REQ-001',
  check: (req, res) => {
    try {
      // logic here
      return match ?? null;
    } catch {
      return null; // never throw
    }
  }
};
```

### Sensitive Value Masking
```typescript
// Always use this before storing or displaying evidence
function maskValue(value: string): string {
  if (value.length <= 8) return '****';
  return value.slice(0, 4) + '*'.repeat(value.length - 8) + value.slice(-4);
}
// "sk_live_abc123xyz789" → "sk_l************z789"
```

## What NOT to Do

- **Never store unmasked secrets, tokens, or passwords** in SQLite or MMKV
- **Never auto-bypass certificate pinning** — detect and flag only
- **Never log request bodies to console** in production builds
- **Never commit CA private keys** to version control
- **Never attempt to repackage third-party apps** as part of the core app flow
- **Do not add MCP servers mid-session** without updating `.mcp.json` and this file

## Architecture Decisions (Already Made — Don't Re-debate)

- React Native (not Flutter) — chosen for JS ecosystem and faster UI iteration
- Local-only storage (no cloud sync) — security tool, data stays on device
- Rule engine in JS thread (not native) — simpler to extend, fast enough for traffic volume
- HAR export format — interoperable with Burp, Postman, browser DevTools
- No root / no jailbreak required — core product constraint, never compromise this

4. Sensitive data is masked in all outputs
5. The feature is visible in the UI with correct severity color coding
